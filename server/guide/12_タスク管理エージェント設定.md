# 12. タスク管理エージェント兼思考ログシステムのセットアップ

## 概要

設計: `../plan/12_task-management.md`。GitHub Project連携部分を、ユーザー自身の手で用意する
必要がある部分の手順書。すべて完了すると自宅サーバー上で`docker compose --profile task-agent`
から動かせるようになり、systemd timerで1日2回自動実行できる。

LLMは、返信パース(job_collect)には自宅サーバー上のOllama(`llama3.2:3b`)を使い、
レコメンドの判断(daily-plan生成・週次蒸留)には指示追従の強いGemini(LiteLLM経由)を使う
(理由は後述「レコメンドの自己強化ループ」の注記)。朝のDiscordレコメンドは夜間に生成した
プランを読むだけ。Discordはbutler-botとは別の**専用Bot**を新規作成する（同じBot名で
発言が混在すると、どのAIが喋っているか分かりにくいため）。

所要時間の目安: 40分程度

| セクション | 必須度 |
|---|---|
| 1. GitHub（Project・リポジトリ・PAT） | 必須 |
| 2. Discord Bot作成 | 必須 |
| 3. .env設定・ローカル動作確認 | 必須 |
| 4. systemd timerへの登録 | 必須（自動実行させる場合） |

## 1. GitHub

### 1.1 個人Organizationの確認・作成

すでに個人用のGitHub Organizationがあればそれを使う。なければ
https://github.com/organizations/new から作成（Free planで十分）。

### 1.2 `personal-tasks` リポジトリの作成

個人Organization配下に、コードを伴わない雑務issueを格納するリポジトリを作成する。

```bash
# gh CLIがある場合
gh repo create <個人Org>/personal-tasks --private
```

なければ https://github.com/new から手動作成でもよい（Private推奨）。

### 1.3 GitHub Project (V2) の作成

1. 個人Organizationのトップページ → **Projects** タブ → **New project**
2. テンプレートは「Board」を選択
3. プロジェクト名は任意（例: `Tasks`）
4. プロジェクトURL末尾の数字が `GITHUB_PROJECT_NUMBER`（例: `.../orgs/xxx/projects/3` なら `3`）

### 1.4 Statusフィールドの選択肢を確認

デフォルトで `Status` フィールド（Todo / In Progress / Done）が作られている。
使う選択肢名を控えておく（Ollamaが返信をパースする際にこの名前を使う）。
変更した場合は `.env` の `GITHUB_STATUS_FIELD_NAME` も合わせる。

### 1.5 既存issueをProjectへ追加

研究室Organizationの既存issue、既存の趣味リポジトリのissue、`personal-tasks`の新規issueを、
それぞれのissueページ右サイドバー「Projects」から今作ったProjectへ追加する
（研究室リポジトリのissue自体は研究室Organization側に残したまま、Projectへの追加だけ行う）。

### 1.6 ラベル設計

各リポジトリで `research` / `personal` / `hobby:プロジェクト名` 等のラベルをissueに付けておく
（`.env`の`TASK_LABELS`で絞り込む場合に使う。空なら全ラベル対象）。

### 1.7 Fine-grained Personal Access Token の発行

1. https://github.com/settings/personal-access-tokens/new にアクセス
2. **Resource owner**: 個人Organization
3. **Repository access**: 対象リポジトリを個別に選択（研究室リポジトリは研究室Organization側の
   トークンになるため、個人Organization発行のPATでは選べない場合がある → その場合は研究室
   Organizationのリポジトリ設定でこのPATのOrganizationからのアクセスを許可する必要がある）
4. **Permissions**:
   - Repository permissions → `Issues`: Read and write
   - Organization permissions → `Projects`: Read and write
5. **Expiration**: 90日（ローテーション運用。期限が来たら再発行して`.env`の`GITHUB_TOKEN`を更新する）
6. 発行されたトークンをコピー（一度しか表示されない）

## 2. Discord Bot作成

butler-bot（`03_Discord_Bot作成.md`）とは**別の新規Bot**を作る。task-agentはREST APIのみ使い
常時接続はしないため、butler-botの手順よりも設定は少ない（Message Content Intentは不要）。

### 2.1 通知用チャンネルの作成

butler-botがいるDiscordサーバー（別サーバーでもよい）に、通知用チャンネルを作成する
（例: `#task-agent`）。

### 2.2 Bot Applicationの作成

1. https://discord.com/developers/applications → **New Application**
2. 名前: `task-agent`（任意。Discord上での表示名になるので、butler-bot=Geminiと区別しやすい
   名前にしておくと後で見て分かりやすい。例: `task-agent`, `llama-secretary`等）
3. 左メニュー **Bot** → **Reset Token** → 表示されたトークンをコピー
   （一度しか表示されない。`server/.env`の`DISCORD_TOKEN_OLLAMA`として使う）

### 2.3 サーバーへの招待

1. 左メニュー **OAuth2** → **URL Generator**
2. SCOPES: `bot`
3. BOT PERMISSIONS: `Send Messages`, `Create Public Threads`, `Read Message History`
4. 生成されたURLをブラウザで開き、2.1で使うサーバーに招待

### 2.4 チャンネルIDの取得

開発者モードでチャンネルを右クリック→「チャンネルIDをコピー」→
`server/.env`の`DISCORD_CHANNEL_OLLAMA`として使う

## 3. .env設定・ローカル動作確認

```bash
cd ~/dotfiles/server
sudo mkdir -p /mnt/data/ai/task-agent
sudo chown -R "$(whoami):$(whoami)" /mnt/data/ai/task-agent

cp task-agent/.env.example task-agent/.env
nano task-agent/.env
```

`task-agent/.env`に埋める値（Discord関連は2で`server/.env`に設定済みなので、
`task-agent/.env`では空でよい）:

| 変数 | 値 |
|---|---|
| `GITHUB_TOKEN` | 1.7で発行したPAT |
| `GITHUB_PROJECT_OWNER` | 個人OrganizationのログインID |
| `GITHUB_PROJECT_NUMBER` | 1.3で確認した番号 |
| `OLLAMA_MODEL` | 既に`llama3.2:3b`を導入済みなら変更不要 |

`server/.env`側に、2で設定した`DISCORD_TOKEN_OLLAMA`と`DISCORD_CHANNEL_OLLAMA`が
入っていることを確認しておく（docker-compose.ymlがここから`DISCORD_TOKEN`/
`DISCORD_CHANNEL_ID`としてtask-agentコンテナに渡す）。

Ollamaに`llama3.2:3b`がまだ入っていない場合:

```bash
docker exec ollama ollama pull llama3.2:3b
```

動作確認（docker-compose経由、本番と同じ実行方法）:

```bash
docker compose --profile task-agent build task-agent-recommend task-agent-collect
docker compose --profile task-agent run --rm task-agent-recommend
```

Discordの`#task-agent`にレコメンドが投稿され、スレッドが立てば成功。スレッドに適当に返信してから：

```bash
docker compose --profile task-agent run --rm task-agent-collect
```

を実行し、GitHub Project側のStatusが更新されるか確認する。
`/mnt/data/ai/task-agent/<日付>.json`にログが保存されているのも確認しておくとよい。

## 4. systemd timerへの登録（自動実行）

```bash
# 4系統をまとめて配置(recommend=朝のDiscordレコメンド, collect=夜の返信回収,
# daily-plan=毎朝4時にワニ博士アプリ用の当日プラン生成, distill=週次で学習メモ更新)
sudo cp scripts/systemd/task-agent-*.service /etc/systemd/system/
sudo cp scripts/systemd/task-agent-*.timer /etc/systemd/system/

sudo systemctl daemon-reload
sudo systemctl enable --now task-agent-recommend.timer
sudo systemctl enable --now task-agent-collect.timer
sudo systemctl enable --now task-agent-daily-plan.timer
sudo systemctl enable --now task-agent-distill.timer
```

確認:

```bash
systemctl list-timers | grep task-agent
```

デフォルトの実行時刻(Asia/Tokyo):

| timer | 時刻 | 役割 |
|---|---|---|
| `task-agent-recommend` | 毎日 07:00 | 夜間の`daily_plan.json`を読んでDiscordへ投稿(無ければOllama生成にフォールバック) |
| `task-agent-collect` | 毎日 22:00 | Discordの返信を回収しStatus更新 |
| `task-agent-daily-plan` | 毎日 04:00 | ワニ博士アプリの「朝の作戦会議」用に当日プラン(`daily_plan.json`)をitem_id付きで生成(Gemini経由、LiteLLM不調時はOllamaへ)。前日の提案 vs 実績を`feedback/recommend_log.jsonl`に記録 |
| `task-agent-distill` | 毎週月 03:30 | 過去2週間のレコメンド実績を蒸留し`learned_prefs.md`を更新(日次プランより前に走らせる) |

時刻を変える場合は`scripts/systemd/task-agent-*.timer`の`OnCalendar`を編集し、
`sudo systemctl daemon-reload`してから該当timerを再起動する。

### レコメンドの自己強化ループ

`daily-plan`が毎朝「何を提案したか」を`daily_plan/<日付>.json`に残し、翌朝その提案が
実際に「今日やる」で承認されたか・**どのタスクを完了したか**を突き合わせて
`/mnt/data/ai/context/feedback/recommend_log.jsonl`に追記する。`distill`が週1でこれを
蒸留して`/mnt/data/ai/context/learned_prefs.md`(この人の傾向メモ)を作り、日次プラン生成と
ワニ博士アプリのライブ生成の両方のプロンプトに注入される。モデルの実再学習は行わず、
in-contextでレコメンドを本人へ寄せていく仕組み。

> **生成モデルについて:** 日次プランと蒸留の判断は当初ローカルの`llama3.2:3b`で試したが、
> プロフィールの「院試最優先」等のルールすら守れず、urgencyも横並びで実用にならなかった
> (サーバRAMの空きが乏しく7B級への引き上げも不可)。そのため判断系はwani/執事Botと同じ
> Gemini(LiteLLM経由)を主経路にしている。1日1回＋週1回程度なのでコストは無視できる。
> `LITELLM_MASTER_KEY`未設定やLiteLLM不調時は自動でOllamaにフォールバックする。
> 朝のDiscordレコメンド(`task-agent-recommend`)も、この夜間プラン(`daily_plan.json`)を
> 読んで投稿するだけにしてある(毎朝Ollamaで作り直さず計算を節約)。当日プランが無い/壊れて
> いるときだけ、その場でOllama生成にフォールバックする。

**フィードバックの粒度と分類:**
- ワニ博士アプリはタスクを完了(Done / ToDo討伐)した瞬間に、`item_id`・タイトル・
  `repo`・`labels`を`wani_state.json`の`history[日付].done_items`にスナップショットする
  (Doneになるとタスクは翌朝GitHubのopen一覧から消え、labels/repoを引けなくなるため)。
- これにより「提案 → 承認 → 実際に完了」がタスク単位で追える(承認は元々タスク単位、
  完了もタスク単位になった)。
- `distill`は完了タスクを`repo`・`labels`・タイトルを手がかりにカテゴリ
  (院試勉強/研究/研究室の雑務/アルバイト/ロボコン/趣味/その他)へ分類し、カテゴリごとの
  承認・完了傾向を学ぶ。カテゴリは`server/.env`の`DISTILL_CATEGORIES`で変更可
  (Google Calendar等ヒントの乏しいものは「その他」に落ちる)。

手動で回すには:

```bash
docker compose --profile task-agent run --rm task-agent-daily-plan
docker compose --profile task-agent run --rm task-agent-distill
```

### 進捗ステートからのタスク創出と時間割(スケジューリング)

GitHub issueにはしないが管理したい領域(院試勉強・研究・バイト等)の進捗を
`/mnt/data/ai/context/projects/<領域>.md`(**git管理外**、テンプレは`server/context/projects.example/`)
に置くと、夜間daily-planがそれを読んで**次にやるべき具体タスクを創出**する。例: 院試なら
過去問の得点率・学習到達度・弱点から「電磁気の境界値問題を過去問3年分(90分)」を生成。

さらに、`server/context/profile.md`の**「生活リズム」節**(起床/就寝/食事/固定予定)と、
その日のGoogleカレンダー予定(n8n)を踏まえ、**空き時間に起床〜就寝の時間割**
(`proposed_schedule`)を提案する。`daily_plan.json`に `generated_tasks` /
`proposed_schedule` / `wake_time` として保存され、Discord朝レコメンド・ワニ博士アプリ・
執事の朝ブリーフィングに出る。

進捗の更新とカレンダー登録は執事Discord bot経由:
- 「過去問2020の電磁気7割だった」→ `update_progress` が `projects/院試.md` に追記
- 「今日の予定入れといて」または `!schedule` → `apply_daily_schedule` が時間割の学習/作業
  タスクだけを(食事/休憩/既存予定は除外)**ワンタップ承認でGoogleカレンダーに一括登録**

> 日次プランは picks+タスク創出+時間割まで出すため出力が大きく、litellm既定の
> `request_timeout: 30` では足りない。`llm_client._gemini_chat_json` が per-request で
> `timeout=90`(env `JUDGE_LLM_TIMEOUT`)を渡して上書きしている。

## トラブルシューティング

| 症状 | 確認箇所 |
|---|---|
| GitHub Projectのissueが1件も取れない | `GITHUB_PROJECT_OWNER_TYPE`（organization/user）、PATのRepository access範囲 |
| Discordに投稿されない | `server/.env`の`DISCORD_TOKEN_OLLAMA`/`DISCORD_CHANNEL_OLLAMA`が正しいか、task-agent用Botがそのサーバー/チャンネルに招待済みか(2.3) |
| butler-botが起動しない/落ちる | `ai/src/bot.py`は`DISCORD_TOKEN_BUTLER`を読む。`server/.env`に旧名`DISCORD_TOKEN`のまま残っていないか確認 |
| Statusが更新されない | Ollamaが出力した`status`名が実際のフィールド選択肢と完全一致しているか（`GITHUB_STATUS_FIELD_NAME`）。`llama3.2:3b`は時々フォーマットを外すので、`/mnt/data/ai/task-agent/<日付>.json`の`parsed_updates`を見て挙動を確認する |
| `docker compose run`で権限エラー | `/mnt/data/ai/task-agent`の所有者が`kokko`になっているか |
| systemd timerが起動しない | `systemctl status task-agent-recommend.timer`、`journalctl -u task-agent-recommend.service` |
| Ollamaの応答が遅い/タイムアウトする | `llm_client.py`はissue・返信ごとに個別にOllamaを呼ぶため件数が多いと時間がかかる。急ぎではないので基本は待てばよいが、頻発する場合は`requests`のtimeout(180秒)を延ばす |
