# 06. n8n ワークフロー設定

## n8n とは

GUIでワークフローを組める自動化ツールです。
「毎朝8時にGoogleカレンダーを見てDiscordに通知する」といったことを
プログラミングなしで設定できます。

---

## ステップ1: n8n にログイン

http://kokko-server-pavilion:5678 にアクセス（Tailscale接続必須）

- ユーザー名: `kokko`
- パスワード: `.env` の `N8N_PASSWORD`

> HTTPSではなくHTTPでアクセスしているため、n8nの「secure cookie」エラーが出る場合がある。
> `docker-compose.yml`で`N8N_SECURE_COOKIE=false`を設定済み（詳細: `09_トラブルシューティング.md`）。
> （Googleカレンダー連携のOAuthだけは下記ステップ2でHTTPS化が別途必要。ログイン自体はHTTPのままでよい）

---

## ステップ2: Googleカレンダー連携の設定

GoogleはOAuthのリダイレクトURIに①パブリックTLD配下のドメイン、②HTTPS、の両方を要求する。
そのためカレンダー連携だけは`tailscale serve`でHTTPS化してから設定する。

### 2-0. Tailscale serveでHTTPS化（サーバー上で一度だけ実行）

```bash
ssh kokko@kokko-server-pavilion
sudo tailscale serve --bg 5678
tailscale serve status   # マッピングを確認
```

これで `https://kokko-server-pavilion.tailed0412.ts.net`（443番、パス省略）が
`http://127.0.0.1:5678`（n8n）に転送されるようになる。ブラウザで開いて証明書エラーなく
n8nのログイン画面が出ればOK（設定はtailscaledに保存され再起動後も維持される）。

> `tailscale serve`実行時に証明書関連のエラーが出た場合、Tailscale管理コンソール
> (https://login.tailscale.com/admin/dns) の「HTTPS Certificates」がOFFになっている可能性がある。
> ONにしてから再実行する。

### OAuth2認証の設定

1. 右上メニュー → **「Settings」** → **「Credentials」**
2. **「Add Credential」** → **「Google Calendar OAuth2 API」**
   （見当たらない場合は、ワークフロー上に「Google Calendar」ノードを追加し、ノード内の
   Credentialドロップダウンから「Create New Credential」を選ぶと確実）
3. 以下の情報を入力（Google Cloud Consoleで取得）：

#### Google Cloud Console での設定

1. https://console.cloud.google.com にアクセス
2. プロジェクトを作成（例: homeserver）
3. **「APIとサービス」** → **「ライブラリ」** → 「Google Calendar API」を有効化
4. **「OAuth同意画面」** を設定：
   - ユーザーの種類: **外部**
   - アプリ名: homeserver
   - テストユーザーに自分のGmailを追加
5. **「認証情報」** → **「認証情報を作成」** → **「OAuthクライアントID」**
   - アプリケーション種類: **ウェブアプリケーション**
   - 承認済みの JavaScript 生成元: **空欄のまま**（n8nはサーバー側の認可コードフローのため不要。
     入れると「無効な生成元」エラーになる）
   - 承認済みのリダイレクトURI: `https://kokko-server-pavilion.tailed0412.ts.net/rest/oauth2-credential/callback`
     > - HTTP（ポート5678）のままだと最終的にGoogleログイン時に
     >   「Error 400: invalid_request / doesn't comply with Google's OAuth 2.0 policy」で弾かれる
     >   → 上記2-0でHTTPS化した443番のURLを使う（ポート番号は書かない）
     > - 短縮ホスト名（`kokko-server-pavilion`のみ）だと「末尾はパブリックトップレベルドメインが
     >   必要」エラーになる → TailscaleのMagicDNSサフィックスまで含めたフルFQDNを使う
     > - `docker-compose.yml`の`WEBHOOK_URL`もこのURLに合わせて設定済み（詳細: `09_トラブルシューティング.md`）
6. クライアントIDとシークレットをコピー

n8nの認証情報画面に入力 → 「OAuth Redirect URL」欄が上記のhttps URLになっていることを確認
→ **「Sign in with Google」** → Googleアカウントで認証

## ステップ3: 毎朝のデイリーブリーフィング ワークフロー

> **実際にはこの形では実装していない（2026-07-08時点）。** 以下はn8n単体で完結させる場合の
> 参考手順として残してあるが、実際に稼働しているのは「n8nはGoogleカレンダーの取得だけ担当し、
> Discordへの投稿はbutler-bot（Gemini執事）が行う」という構成。理由は、Discord送信は既に
> butler-botが担っており（起動通知・チャットなど）、n8n側にも別途Discord連携を作ると窓口が
> 二重になるため（[[task-management-design-preference]]の「既存インフラ再利用」方針）。
>
> **実装済みの構成:**
> - n8nに **Webhookトリガー**のワークフロー「毎朝ブリーフィング_今日の予定取得」を作成済み
>   （ワークフローID: `y9mTvYxXOjj3UzrX`、Publish済みで有効化状態）
>   - `Webhook`（GET `/webhook/today-schedule`、認証なし・tailnet内のみ運用前提）
>     → `カレンダー一覧`（Codeノード。対象カレンダーIDを1件ずつitemとして流す。
>     **2026-07-11から複数カレンダー対応**: 自分のprimary + chiso.ships@gmail.com +
>     グループカレンダー2つ。追加/削除はこのノードのIDリストを編集）
>     → `Google Calendar`（Get Many、Calendar: `{{ $json.calendarId }}`のID指定、
>     After: `{{ $today }}`、Before: `{{ $today.plus({ days: 1 }) }}`。
>     **注意: ID指定モードで`primary`という値はバリデーションで弾かれる**ため、
>     自分のカレンダーも実ID(kogakou.k@gmail.com)で書くこと）
>     → `重複除去`（Codeノード。同じ予定が複数カレンダーに載っている場合に
>     iCalUIDで1つにする）→ `Respond to Webhook`（Respond With: All Incoming Items）
>   - 動作確認: `curl http://127.0.0.1:5678/webhook/today-schedule`（サーバー上、またはdocker
>     composeネットワーク内から`http://n8n:5678/webhook/today-schedule`）
> - `docker-compose.yml`の`butler-bot`サービスに`N8N_TODAY_SCHEDULE_URL`を追加し、
>   `ai/src/bot.py`に`fetch_today_schedule()`を実装。`daily_briefing()`（毎朝8時）がこれを呼んで
>   Discordの`#butler-notify`に投稿する
> - `ai/src/agent_tools.py`に`get_today_schedule`ツールも追加済みなので、チャットで
>   「今日の予定は?」と聞いても同じ経路（n8n Webhook）でGeminiが即答できる
>
> n8nワークフローをUIで作り直す/移行する場合は、CLIインポート(`n8n import:workflow`)で
> ワークフローJSONに**トップレベルの`id`フィールドが無いと`SQLITE_CONSTRAINT: NOT NULL`で
> 失敗する**点、`--userId`を指定すると既存クレデンシャルの所有権再割り当て処理が走り
> `credential with ID "undefined"`という無関係なエラーで失敗する点（`--userId`を省略し、
> 個人プロジェクトに自動で紐付くのを利用するのが安全）に注意。また新規importしたワークフローは
> `active`列がfalseで作られ、`n8n publish:workflow --id=<id>`で有効化しても**n8n本体プロセスを
> 再起動するまでWebhookが登録されない**（`docker compose restart n8n`が必要）。

### 作成するワークフロー（n8n単体で完結させる場合の参考手順）

```
[Schedule: 毎日8:00] → [Google Calendar: 今日の予定取得] → [Discord: 通知送信]
```

### 手順

1. n8nで **「New Workflow」** をクリック
2. **「+」** → **「Schedule Trigger」**
   - Trigger Rule: `Cron`
   - 設定: `0 8 * * *`（毎日8時）
3. **「+」** → **「Google Calendar」** ノードを追加
   - Operation: `Get Many`
   - Calendar: メインカレンダー
   - Time Min: `{{ $now.startOf('day').toISO() }}`
   - Time Max: `{{ $now.endOf('day').toISO() }}`
4. **「+」** → **「Code」** ノードで整形：
   ```javascript
   const events = $input.all();
   if (events.length === 0) {
     return [{ json: { message: "今日の予定はありません。" } }];
   }
   const list = events.map(e => {
     const start = e.json.start?.dateTime || e.json.start?.date;
     return `・${start} - ${e.json.summary}`;
   }).join('\n');
   return [{ json: { message: `📅 **今日の予定**\n${list}` } }];
   ```
5. **「+」** → **「Discord」** ノード
   - Webhook URL: Discord の `#butler-notify` チャンネルのWebhook URL
   - Content: `{{ $json.message }}`

### Discord Webhook URLの取得

1. `#butler-notify` チャンネルを右クリック → **「チャンネルの編集」**
2. **「連携サービス」** → **「ウェブフック」** → **「新しいウェブフック」**
3. URLをコピー

## ステップ4: ディスク残量アラート

```
[Schedule: 毎日12:00] → [Execute Command: df] → [IF: 残量 < 10%] → [Discord: アラート]
```

1. Schedule Trigger: `0 12 * * *`
2. **「Execute Command」** ノード:
   ```bash
   df /mnt/data --output=pcent | tail -1 | tr -d ' %'
   ```
3. **「IF」** ノード:
   - Value 1: `{{ $json.stdout }}`
   - Operation: `Larger`
   - Value 2: `90`
4. **「Discord」** ノード（TRUE側）:
   - Content: `⚠️ ストレージ残量が10%を切りました！`
