### Tailscale接続後、ラップトップのインターネットが全部切れた（LANは生きている）

**原因1: Exit Nodeが有効になっている**

ラップトップ側のTailscaleクライアントで「Use exit node」がONだと、全通信（DNS含む）がサーバー経由になる。
このサーバーはExit Nodeとして構成していない（IPフォワーディング・NAT未設定）ため、通信が行き止まりになる。

対処：ラップトップのTailscaleアプリでExit Nodeを「None」に戻す。

```bash
# CLIがあれば一発で解除
tailscale set --exit-node=
```

このサーバーではExit Node機能は不要（`plan/02_network.md`参照）。ラップトップ側で誤って選択しないよう注意する。

**原因2: 管理コンソールの「Override local DNS」が有効**

https://login.tailscale.com/admin/dns の Nameservers 設定で「Override local DNS」がONになっていると、
tailnet全体のDNSクエリが指定されたネームサーバーに向けられる。そのネームサーバーが機能していないと、
Tailscale接続中は外部サイトが一切開けなくなる（LAN内の直接アクセスは無関係なので生きている）。

**切り分け方法**：接続中にIPアドレス直打ち（`ping 8.8.8.8`等）が通るならDNSの問題、通らないならルーティングの問題（広すぎるSubnet Routesを自動受け入れている可能性）。

対処：管理コンソールで「Override local DNS」をOFFにする。MagicDNS（ホスト名解決）は別トグルなので影響しない。

### MagicDNSでホスト名（`server`等）にアクセスできない

`tailscale status`に表示される名前は、ガイドや過去のメモに書いてある名前（`server`）ではなく、
**そのマシンが実際にTailscaleへ登録された名前**（例：`kokko-server-pavilion`）。

```bash
tailscale status   # 実際のホスト名を確認
resolvectl query <実際のホスト名>
```

`server`のような短い名前で統一したい場合は、サーバー側でホスト名を変更する。

```bash
sudo tailscale up --hostname=server
```

または管理コンソールのMachinesページで対象マシンを選び「Edit machine name」。

### Immich にアクセスできない

```bash
# コンテナが動いているか確認
sudo docker ps | grep immich

# PostgreSQLが起動するまで待つ（初回は2〜3分かかる）
sudo docker compose logs -f database

# ポートの確認
sudo ss -tlnp | grep 2283
```

**「connection refused」になる場合（ポート番号は空いているのに拒否される）**

`docker ps`で`immich_server`が`unhealthy`や再起動を繰り返している（`Up 1 second`のような短い稼働時間）状態なら、
まずログでエラー内容を確認する。

```bash
sudo docker logs --tail=80 immich_server
```

- `PostgresError: password authentication failed for user "immich"` が出る場合：
  `.env`の`DB_PASSWORD`を初回セットアップ後に変更・再生成すると、Postgres内部には初回起動時のパスワードが
  残ったままになり、Immichサーバーが新しいパスワードで繋ぎに行って認証失敗する。
  データを消さずにパスワードだけ揃える：
  ```bash
  cd ~/dotfiles/server
  source .env
  sudo docker exec -it immich_postgres psql -U immich -d immich -c "ALTER ROLE immich WITH PASSWORD '$DB_PASSWORD';"
  sudo docker restart immich_server immich_microservices
  ```

- コンテナが安定して起動しているのに`connection refused`になる場合：
  `docker-compose.yml`のポートマッピング（`ports:`）とImmichイメージが実際にリスニングしているポートが
  ズレていないか確認する。Immichはバージョンによって内部リスニングポートが変わることがある
  （v3系では`2283`、旧バージョンでは`3001`だった）。
  ```bash
  sudo docker logs immich_server | grep "listening on"
  ```
  ログの`http://[::1]:<port>`と`docker-compose.yml`の`ports: "2283:<port>"`が一致しているか確認し、
  ズレていれば`docker-compose.yml`側を修正して`sudo docker compose up -d immich-server`で再作成する。

### `docker-compose.yml`に`mem_limit`を設定した後、Node.js系サービスがクラッシュループする

n8nで発生した実例：`mem_limit: 512m`を設定した直後、`docker inspect n8n`の`RestartCount`が
15〜20秒間隔で増え続け、ログに以下が出る。

```
FATAL ERROR: Ineffective mark-compacts near heap limit Allocation failed - JavaScript heap out of memory
```

**原因**：Node.js（v18以降）はcgroupのメモリ上限を自動検出してV8のヒープサイズを決める。
`mem_limit`を小さく設定すると、V8が計算する上限（`heap_size_limit`）も連動して小さくなり、
実際のアプリの必要量に対して不足してクラッシュする。確認方法：

```bash
docker exec <container> node -e 'console.log(require("v8").getHeapStatistics().heap_size_limit / 1024 / 1024, "MB")'
```

また`cat /sys/fs/cgroup/system.slice/docker-<container id>.scope/memory.events`で
`oom_kill: 0`ならカーネルのOOM Killerではなく、V8自身がヒープ上限に達して自滅している。

**対処**：`mem_limit`を上げつつ、V8のヒープ上限を明示的に指定する（自動計算に任せない）。

```yaml
    environment:
      - NODE_OPTIONS=--max-old-space-size=768   # mem_limitの70〜75%程度を目安に
    mem_limit: 1024m
```

適用は`docker compose up -d <サービス名>`（`sudo`不要、`docker`グループに入っていれば実行可）。

### samba（smbd）がSMB接続を受け付けない・`unhealthy`が続く（解決済み）

**原因**：`smbd -F --debug-stdout --no-process-group -d 5`で手動起動して詳細ログを見ると、

```
directory_create_or_exist_strict: invalid ownership on directory /var/cache/samba/msg.lock
```

（や `/var/lib/samba/private/msg.sock`）で即終了していた。`/var/cache/samba`と`/var/lib/samba`は
実は`/data/cache`・`/data/lib`（＝ホストの`/mnt/data`配下）へのシンボリックリンクで、
これらのディレクトリが過去の設定（uid 1000）のまま残っていたのに対し、smbdは`root(uid 0)`として
起動するようになっていたため所有者不一致でSambaの厳格なセキュリティチェックに引っかかっていた。
メモリ（`mem_limit`）とは無関係だった。

**対処**：

```bash
docker exec samba readlink -f /var/cache/samba   # → /data/cache
docker exec samba readlink -f /var/lib/samba     # → /data/lib
docker exec samba chown -R root:root /data/cache /data/lib
docker restart samba
```

`chown -R`にシンボリックリンクをそのまま渡すと**リンク自体**だけが変わり中身は変わらないので、
`readlink -f`で実体パスを確認してからそちらに対して実行すること。

### OpenWebUIで「no model available」になる／接続を追加したのに反映されない

OpenWebUIは初回起動時こそ`OPENAI_API_KEY`・`OPENAI_API_BASE_URL`・`OLLAMA_BASE_URL`等の環境変数を
読むが、**一度DB（`openwebui-data`ボリューム内の`webui.db`）に接続設定が保存されると、
以後は環境変数を無視してDBの値を使い続ける**。そのため`docker-compose.yml`の環境変数を変更して
コンテナを再作成しても、古い接続先やAPIキーがそのまま使われてしまうことがある。

**確認方法**（値そのものは表示しない。ログに`External Error: ... Please pass a valid API key`や
`Cannot connect to host host.docker.internal:11434`のようなエラーが出ていないか確認）：

```bash
docker logs --tail 60 openwebui | grep -iE "model|ollama|openai|error"
```

**対処**：管理画面（Settings → Connections）から手動で直すのが正規の方法。DBを直接触る場合は
（DBの生の値は画面に表示しないこと）：

```bash
docker exec openwebui python3 -c "
import sqlite3, json, os
conn = sqlite3.connect('/app/backend/data/webui.db')
cur = conn.cursor()
cur.execute(\"UPDATE config SET value=? WHERE key='ollama.base_urls'\", (json.dumps(['http://ollama:11434']),))
key = os.environ['OPENAI_API_KEY']  # コンテナに渡っている環境変数から取得、値は出力しない
cur.execute(\"UPDATE config SET value=? WHERE key='openai.api_keys'\", (json.dumps([key]),))
conn.commit()
print('updated, key length:', len(key))
"
docker restart openwebui
```

### n8n のデータが消えた

n8nのデータは `/mnt/data/ai/n8n` に保存されています。

```bash
ls -la /mnt/data/ai/n8n/
```

`docker compose down -v` を実行するとボリュームが消えますが、
n8nのデータはホストのディレクトリに保存されているため安全です。
