# 09. トラブルシューティング

## よくある問題と解決法

---

### サービスが起動しない

```bash
# 全サービスの状態確認
sudo docker compose ps

# エラーログを確認
sudo docker compose logs --tail=50 <サービス名>
# 例: sudo docker compose logs --tail=50 immich-server
```

---

### Immich にアクセスできない

```bash
# コンテナが動いているか確認
sudo docker ps | grep immich

# PostgreSQLが起動するまで待つ（初回は2〜3分かかる）
sudo docker compose logs -f database

# ポートの確認
sudo ss -tlnp | grep 2283
```

---

### n8n のデータが消えた

n8nのデータは `/mnt/data/ai/n8n` に保存されています。

```bash
ls -la /mnt/data/ai/n8n/
```

`docker compose down -v` を実行するとボリュームが消えますが、
n8nのデータはホストのディレクトリに保存されているため安全です。

---

### Butler Bot が返事をしない

```bash
# ログを確認
sudo docker compose logs -f butler-bot

# よくあるエラー:
# DISCORD_TOKEN が無効 → Discord Developer Portalでトークンを再発行
# GEMINI_API_KEY が無効 → Google AI Studioで確認
# インテントが無効 → Message Content Intent をONに
```

---

### Samba に接続できない

```bash
# Sambaコンテナのログ
sudo docker compose logs samba

# ポート445が開いているか
sudo ss -tlnp | grep 445

# UFWでSambaが許可されているか
sudo ufw status | grep 445
# → 許可されていない場合:
sudo ufw allow 445/tcp
```

---

### /mnt/data がマウントされていない

再起動後にマウントされない場合：

```bash
# 手動マウント
sudo mount /mnt/data

# fstabの確認
cat /etc/fstab | grep data

# ディスクの状態確認
lsblk -f /dev/sdb
```

---

### RAM使用量が高い

```bash
# メモリ使用状況
free -h
sudo docker stats --no-stream

# Immichの機械学習コンテナを停止して節約
sudo docker stop immich_machine_learning
# または docker-compose.yml で profiles: ["ml"] を追加して再起動
```

---

### Docker が起動しない

```bash
sudo systemctl status docker
sudo systemctl start docker
sudo systemctl enable docker  # 自動起動設定
```

---

## ディスク残量確認

```bash
df -h /mnt/data
# または
du -sh /mnt/data/*
```

---

## 全サービスの再起動（最終手段）

```bash
cd ~/dotfiles/server
sudo docker compose down
sudo docker compose up -d
```

---

## ログ一括確認

```bash
cd ~/dotfiles/server
sudo docker compose logs --tail=20
```
