# ダッシュボード表示 + サーバー本体での音声対応（提案）

## 背景

`05_voice.md`は「スマホから声で命令→サーバーで処理→スマホやDiscordで返答」という設計で、
マイク・スピーカーは**スマホ側**にある前提だった。

今回のリクエストは以下の2点で、媒体は**サーバーPC本体**：

1. サーバーPCに何らかの「ホーム画面」的な表示をさせたい
2. サーバーPC自体で音声対応（マイク入力・スピーカー出力）をさせたい

この2つは実機（ディスプレイ・マイク・スピーカーの接続、パッケージインストール、常時稼働プロセスの追加）が
必要になるため、深夜のうちに実装はせず、要件と選択肢の整理のみ行う。
実施の際は「何を・なぜ・想定される影響」を確認してから進める。

---

## 1. ダッシュボード表示

### 前提確認（2026-07-05に判明：想定と異なる）

`00_はじめに.md`や`01_hardware.md`はヘッドレス運用を想定していたが、実際には**このマシンは
GUI付きデスクトップとして常用されている**（`ps aux`で`gnome-shell`・Vivaldiブラウザ・複数の
Claude Codeセッションが常時稼働中を確認）。つまり：

- モニター・入力機器は元から接続されている
- Xorg/Waylandのデスクトップ環境も入っている
- 「モニターを新たに用意する」フェーズは不要で、**既存の画面にKiosk用ブラウザを1枚追加する**だけで
  ダッシュボード表示は技術的に実現可能

ただし、このマシンは**ユーザー自身が日常的に使っているデスクトップ**でもあるため、勝手に
画面を占有・変更する（ブラウザを強制起動する、ディスプレイ設定を変える等）のは、バックエンドの
docker-composeサービス追加とは性質が異なりユーザーの作業を妨げる可能性がある。このため
**GUI操作を伴う実装は今回は見送り、方針の確認のみ行った**（バックエンドのHomepageダッシュボード自体は
`docker-compose.yml`に追加済みで、ブラウザから`http://kokko-server-pavilion:3005`でアクセス可能）。

実施するなら次の一手は「常時表示のKiosk化」ではなく、「ユーザーが好きなタイミングで見るブックマーク」
に留める方が、共用デスクトップとの相性が良い可能性がある。

### 構成案

```
[モニター] ← HDMI/DP ← [サーバーPC]
                           │
                    軽量デスクトップ（Xorg最小構成 or ラベル無しWM）
                           │
                    Chromiumをkiosk起動
                           │
                    ダッシュボードアプリ（Web UI）
```

### ダッシュボードアプリの選択肢

| ツール | 特徴 | メモリ目安 |
|-------|-----|----------|
| **Homepage**（gethomepage.dev） | YAML設定、サービスへのリンク・ステータス表示、Docker統合あり | 〜100MB |
| **Glance** | 軽量、ウィジェット式（時計・カレンダー・RSS等） | 〜50MB |
| **Homarr** | UIが豪華だがやや重い | 〜200MB |
| 自作HTMLページ | 最小構成、静的ページ+JSで各サービスのステータスをfetch | ほぼ0 |

→ 既存のUptime Kuma・Immich・OpenWebUI・n8nへのリンクとステータスを一覧表示する用途なら
**Homepage**が既存運用（Docker Compose・YAML管理）と最も馴染む。

### Kiosk起動の構成案

- 最小構成：`lightdm` + `openbox` + 自動ログイン + `.config/openbox/autostart`で
  `chromium --kiosk --app=http://localhost:<port>` を起動
- systemdサービス化して再起動時も自動復帰

### 未決定事項

- [ ] モニターは接続済みか（サイズ・向き・タッチ対応の有無）
- [ ] 常時画面点灯でよいか（スクリーンセーバー/省電力設定の方針）
- [ ] 表示内容：サービス一覧だけで良いか、カレンダー・天気・執事の一言なども載せたいか
- [ ] Homepage / Glance / 自作のどれで進めるか

## 2. サーバー本体での音声対応

### `05_voice.md`との関係

STT/TTSのバックエンド選定（Google STT/TTS → Phase2でwhisper.cpp/VOICEVOXへ移行）や
Switchbot連携のアーキテクチャは`05_voice.md`をそのまま継承する。
今回の差分は**入出力デバイスの位置**のみ：

| | `05_voice.md`（既存） | 今回の追加要件 |
|---|---|---|
| マイク入力 | スマホ（Tasker等） | サーバー本体に接続したUSBマイク |
| トリガー方式 | ボタン押し発話（スマホ） | 常時待機＋ウェイクワード（スマートスピーカー的運用）|
| スピーカー出力 | スマホで再生 | サーバー本体に接続したスピーカー |

`05_voice.md`の「トリガー方式の選択肢」表に、以下の行を追加する形で統合するのが良い：

| 方式 | 説明 | 難易度 |
|-----|------|--------|
| **サーバー本体常時待機（ウェイクワード）** | サーバーのマイクが「ヘイ執事」等を検出後に起動。スマホ操作不要 | 中〜高 |

### 前提確認が必要な事項（未確認）

- サーバーにUSBマイク・スピーカーは接続済みか
- 常時稼働のウェイクワード検出プロセスがRAM 7.1GB環境で他サービスと共存できるか
  （`openWakeWord`はモデルが小さく常時稼働でも数十〜100MB程度が目安。ただしIntel iGPUのみで
  CPU推論になるため、STT/TTSと同時実行時の負荷は実測が必要）

### 構成案

```
[USBマイク] → 常時ウェイクワード検出（openWakeWordなど、軽量ローカル）
                  │ 検出
                  ▼
            録音開始 → Google STT API（05_voice.md Phase1方針を継承）
                  │
                  ▼
            AIエージェントコア（04_ai-butler.md / Butler Bot）
                  │
                  ▼
            Google TTS API → [USBスピーカー] で再生
```

### 未決定事項

- [ ] ウェイクワードのツール選定（openWakeWord / Porcupine）— `05_voice.md`の未決定事項と同一
- [ ] マイク・スピーカーのハードウェア確保（USB接続の一般的な製品で足りるか、指向性が必要か）
- [ ] 常時プロセスをどのコンテナ/サービスとして常駐させるか（butler-bot拡張か、別サービスとして分離するか）
- [ ] プライバシー面：常時マイクON状態の扱い（ミュートスイッチ・LED表示の検討）
