# AIエージェント（執事）

## コンセプト

単なるチャットボットではなく、生活・作業を能動的にサポートするエージェント。
自分から提案し、外部サービスを操作し、ロボットに命令できる「家の頭脳」。

## ユースケース一覧

### 能動型（エージェントから話しかけてくる）

| ユースケース | 説明 |
|------------|------|
| 行動提案 | Googleカレンダーを見て「10分後にMTGがあります。資料は準備できていますか？」|
| リマインダー | 設定した時間やイベントに応じてDiscordに通知 |
| 状況報告 | 「今日のタスク」「未読メール要約」などを朝まとめて通知 |
| 異常検知 | サーバーダウン・ディスク残量逼迫などをアラート |

### 受動型（こちらから命令する）

| ユースケース | 入力例 | エージェントの動作 |
|------------|------|----------------|
| カレンダー操作 | 「来週月曜に田中さんとMTG入れて」| Google Calendar API で予定作成 |
| 資料作成 | 「明日のゼミ資料のアウトライン作って」| テンプレート生成 → ファイル保存 |
| ファイル整理 | 「downloadsフォルダを整理して」| ファイル一覧を確認→カテゴリ分けして移動 |
| ロボット制御 | 「玄関に行って」| 高レベル命令 → ROS2 アクションに変換 |
| Switchbot操作 | 「エアコンつけて」「電気消して」| Switchbot API 呼び出し |
| 情報検索 | 「昨日保存した論文の要約は？」| ファイルサーバーを検索して返答 |

## インターフェース

### Discord Bot（メインUI）

| 項目 | 内容 |
|-----|------|
| 用途 | テキストチャット・通知の送受信 |
| チャンネル設計 | `#butler-chat`（対話）、`#butler-notify`（能動通知）、`#robot-status`（将来）|
| 呼び出し | `@butler <命令>` または自然文 |
| 通知受信 | スマホの Discord アプリで受け取る |

### OpenWebUI（チャットUI補助）

Tailscale 経由でブラウザから使う本格的なチャットインターフェース。
PC での長い対話や、モデル切替が必要な場面で使用。

## バックエンド構成

```
[Discord Bot / OpenWebUI]
        │
        ▼
[エージェントコア]  ←── ツール群
        │               ├── Google Calendar API
        │               ├── Switchbot API
        │               ├── ファイルシステム操作
        │               ├── ROS2 Bridge（Phase 3）
        │               └── Web検索
        ▼
[LLM バックエンド]
  Phase 1: Gemini API（Google）
  Phase 2: Ollama on GPU PC（Tailscale 経由 http://gpu-pc:11434）
  フォールバック: GPU PC 停止時は Gemini API に自動切替
```

## Phase 1 技術スタック

| コンポーネント | 採用技術 | 理由 |
|-------------|---------|------|
| LLM | Gemini API (gemini-2.0-flash) | クレジット活用、高性能 |
| エージェントフレームワーク | LangGraph または n8n | ツール呼び出し・状態管理 |
| Discord 連携 | discord.py | Python エコシステムと統合しやすい |
| カレンダー | Google Calendar API | Googleクレジットと同じアカウント |
| スケジューラ | APScheduler (Python) または n8n のCronノード | 能動通知のトリガー |

### n8n vs LangGraph の選択基準

| | n8n | LangGraph |
|--|-----|-----------|
| 向き | ワークフロー自動化（IF-THEN型）| 複雑なエージェント（ループ・分岐）|
| UI | ノードエディタ（GUIで設定）| コードベース |
| 難易度 | 低い | 高い |
| 柔軟性 | 中 | 高い |

→ Phase 1 は n8n で始めて、複雑なエージェント挙動が必要になったら LangGraph に移行。

## 能動通知のタイミング設計

完全な「常時監視→自発的判断」は計算コストが高い。
現実的な設計として以下の方式を組み合わせる：

| トリガー種別 | 例 | 実装方法 |
|------------|---|---------|
| 時刻ベース | 「毎朝8時に今日の予定を通知」| Cron |
| イベントベース | 「カレンダーに予定が入ったら」| Calendar API の push通知 or ポーリング |
| 閾値ベース | 「ディスク残量が10%を下回ったら」| 監視スクリプト |
| 手動トリガー | 「今日どうだっけ」とDiscordで聞く | Discord Bot |

## 記憶・コンテキスト管理

エージェントが「昨日話したこと」を覚えておくための設計：

- 会話履歴：SQLite または JSON ファイル（`/mnt/data/ai/context/`）
- 長期記憶：重要な情報をベクトルDBに保存（将来。Phase 2 以降で Chroma 等）
- ユーザープロファイル：好み・習慣をファイルに記録
